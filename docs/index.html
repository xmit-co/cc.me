cc.me
Use cases OAuth Safe inbox JS client

OAuth callbacks. Sealed webhooks.

cc.me redirects OAuth callbacks to local targets and keeps webhook deliveries encrypted until you claim and acknowledge them.

Use cases

  • Local OAuth callbacks. Register one HTTPS redirect URI, then trampoline OAuth responses to a local app like http://example.local/auth/callback while preserving provider query parameters.
  • Stable callback aliases. Create short /c/... URLs for preview apps, branch environments, demos, or providers that make redirect URI changes slow to review.
  • Encrypted webhook capture. Point a provider at /i/<ed25519-public-key>; deliveries wait encrypted until your app claims, handles, and acknowledges them.
  • Local forwarding. Run the JS CLI to claim inbox deliveries and forward them to http://example.local:8080/webhook, acknowledging only after the local handler succeeds.
  • Real-time inspection. Use cc-me inspect, GET cursors, or peek to watch deliveries arrive without removing them.
  • Controlled retries. Claim batches of messages, ack the handled IDs, and release anything that should be retried immediately.
  • Team fanout. Send one provider webhook to multiple public keys so teammates or separate services each get their own encrypted copy.
  • Protocol handshakes. Use adapter URLs for Webmention, WebSub, Slack Events, Pingback, Meta-style webhooks, CloudEvents, and Discord Interactions when providers need endpoint-specific responses.
  • Provider verification. Capture GitHub, GitLab, Stripe, and Shopify deliveries first, then verify their signatures from the decrypted raw body and headers before acting on them.
  • Private test fixtures. Trigger a provider in a test, claim the encrypted delivery, assert the captured request, and acknowledge it when the test passes.
  • Offline handoffs. Capture callbacks for a tool that is not online yet, then share only the private key with the person or process allowed to read them.

OAuth callbacks

Use / with an absolute at target. Every callback query parameter except at is appended to the target. 

https://cc.me/?at=http%3A%2F%2Fexample.local%2Fauth%2Fcallback

Short aliases

Create a path-only callback URL when a provider wants a stable redirect URI. Reusing the same target returns the same alias. 

POST /c

{
  "at": "http://example.local/auth/callback"
}
{
  "url": "https://cc.me/c/1cuEdvnPA31f"
}

Safe inbox

/i/<base64url-ed25519-public-key> is a tiny encrypted inbox.

  • POST captures a request and stores it encrypted for that public key.
  • POST /i/<key-a>.<key-b> fans out one encrypted copy to each recipient.
  • The starting default keeps up to 100 hooks. When an inbox is full, POST is rejected and existing deliveries are left alone.
  • Requests are stored only when the captured method, path, query, headers, and body fit within 64 KiB; larger requests are rejected.
  • GET looks ahead without removing deliveries. Add optional ?l=10 to set the maximum batch size and/or ?p to wait when nothing is ready. Responses include a cursor; pass it back as ?c=... to read newer visible deliveries.
  • Protocol adapters live under the same inbox: /webmention, /websub, /slack, /pingback, /meta, /cloudevents, and /discord/<app-public-key>. 
POST /i/xG72MCGkluShRByISwEzZVgjIUti3_7phnEch9PxJRI

When any recipient inbox is full, the capture is rejected.

HTTP/1.1 409 Conflict
{
  "error": "inbox is full"
}

Protocol adapters

Adapters answer provider handshakes, then store the complete incoming request encrypted. Most signature checks stay with the client after decrypting; Discord requests are verified before capture because Discord requires that at the endpoint. 

POST /i/xG72MCGkluShRByISwEzZVgjIUti3_7phnEch9PxJRI/webmention
content-type: application/x-www-form-urlencoded

source=https%3A%2F%2Fexample.com%2Fpost&target=https%3A%2F%2Fexample.net%2Fpage
GET /i/xG72MCGkluShRByISwEzZVgjIUti3_7phnEch9PxJRI/websub?hub.mode=subscribe&hub.topic=https%3A%2F%2Fexample.com%2Ffeed&hub.challenge=abc
POST /i/xG72MCGkluShRByISwEzZVgjIUti3_7phnEch9PxJRI/slack
content-type: application/json

{
  "type": "url_verification",
  "challenge": "3eZbr..."
}
POST /i/xG72MCGkluShRByISwEzZVgjIUti3_7phnEch9PxJRI/pingback
content-type: text/xml

<methodCall><methodName>pingback.ping</methodName>...</methodCall>
GET /i/xG72MCGkluShRByISwEzZVgjIUti3_7phnEch9PxJRI/meta?v=secret&hub.mode=subscribe&hub.verify_token=secret&hub.challenge=abc
POST /i/xG72MCGkluShRByISwEzZVgjIUti3_7phnEch9PxJRI/cloudevents
ce-specversion: 1.0
ce-id: evt_1
ce-source: https://example.com/source
ce-type: com.example.event
content-type: application/json

{ "ok": true }
POST /i/xG72MCGkluShRByISwEzZVgjIUti3_7phnEch9PxJRI/discord/0123456789abcdef...
x-signature-ed25519: ...
x-signature-timestamp: 1781337600
content-type: application/json

{ "type": 1 }

Meta verification returns the plain hub.challenge. CloudEvents accepts binary HTTP mode and structured JSON. Discord type: 1 returns {"type":1}; other valid interactions are stored and answered with {"type":5}.

Provider recipes

GitHub, GitLab, Stripe, and Shopify can use the normal inbox URL unless you want one of the adapters above. After decrypting, verify the captured raw bodyBytes with the provider headers before acting on the payload.

  • GitHub: check x-hub-signature-256.
  • GitLab: prefer Standard Webhooks headers webhook-id, webhook-timestamp, and webhook-signature; older hooks may use x-gitlab-token.
  • Stripe: check stripe-signature.
  • Shopify: check x-shopify-hmac-sha256.

Delivery control

Claim reserves deliveries for handling. Ack and release both take batches of IDs, but they mean different things:

  • ack means finished. The delivery is removed and will not be returned.
  • release means not finished. The delivery is ready for another try now.
  • If a client disappears after claiming, the service eventually makes it ready again.

Owner authentication

GET, claim, ack, and release must prove ownership of the inbox private key. Each request needs an x-cc-me-timestamp header (Unix seconds) and an x-cc-me-signature header containing a base64url Ed25519 signature of: 

cc-me-v1
{METHOD}
{PATH}
{TIMESTAMP}
{SHA256(BODY) base64url}

The JS client adds these headers automatically.

GET responses

GET looks ahead without removing deliveries. GET and claim responses include only as many items as fit within 1 MiB. 

GET /i/xG72MCGkluShRByISwEzZVgjIUti3_7phnEch9PxJRI?l=10&p
x-cc-me-timestamp: 1781337600
x-cc-me-signature: base64url-ed25519-signature
{
  "count": 1,
  "items": [
    {
      "id": "m_01k8v7xq7gx1a8p0g7nq0f",
      "sealed": "base64url-encrypted-delivery"
    }
  ],
  "cursor": "opaque-cursor"
}

Claim

Claim reserves ready deliveries until you ack or release them.

POST /i/xG72MCGkluShRByISwEzZVgjIUti3_7phnEch9PxJRI/claim
x-cc-me-timestamp: 1781337600
x-cc-me-signature: base64url-ed25519-signature

{
  "limit": 10,
  "poll": true
}
{
  "count": 1,
  "items": [
    {
      "id": "m_01k8v7xq7gx1a8p0g7nq0f",
      "sealed": "base64url-encrypted-delivery"
    }
  ]
}

Ack

Ack removes handled deliveries. Unknown or already-gone IDs are returned as missing.

POST /i/xG72MCGkluShRByISwEzZVgjIUti3_7phnEch9PxJRI/ack
x-cc-me-timestamp: 1781337600
x-cc-me-signature: base64url-ed25519-signature

{
  "ids": ["m_01k8v7xq7gx1a8p0g7nq0f"]
}
{
  "acked": 1,
  "missing": []
}

Release

Release makes claimed deliveries ready again without deleting them.

POST /i/xG72MCGkluShRByISwEzZVgjIUti3_7phnEch9PxJRI/release
x-cc-me-timestamp: 1781337600
x-cc-me-signature: base64url-ed25519-signature

{
  "ids": ["m_01k8v7xq7gx1a8p0g7nq0f"]
}
{
  "released": 1,
  "missing": []
}

After decrypting an item with your private key, the payload has this shape.

{
  "id": "m_01k8v7xq7gx1a8p0g7nq0f",
  "received_at_unix_ms": 1781337600000,
  "method": "POST",
  "path": "/i/...",
  "query": "source=github&delivery=42",
  "headers": [
    {
      "name": "content-type",
      "value_b64u": "YXBwbGljYXRpb24vanNvbg"
    }
  ],
  "body_b64u": "e30"
}

JS client

Forward a Safe inbox to a local endpoint with the cc-me command: 

npx cc-me http://example.local:8080/webhook
pnpx cc-me http://example.local:8080/webhook
bunx cc-me http://example.local:8080/webhook
deno run -A npm:cc-me http://example.local:8080/webhook

Or inspect a Safe inbox locally before acknowledging deliveries:

npx cc-me inspect
pnpx cc-me inspect
bunx cc-me inspect
deno run -A npm:cc-me inspect

The forwarder persists and reuses ~/.cc-me.key by default. Use --key to choose a different key file: 

npx cc-me --key ~/hooks.key http://example.local:8080/webhook

The package works as ESM in browsers, Node.js, Bun, and Deno. In browsers, privateKey() creates an in-memory key, or you can pass a stored key string to new CcMeClient({ privateKey }). In Node.js, Bun, and Deno, privateKey(path) creates a key file private to the user if it does not exist yet, then reuses it on later runs. 

import { homedir } from "node:os";
import { CcMeClient, createAlias, privateKey } from "cc-me";

const alias = await createAlias("http://example.local/auth/callback");
console.log(alias.url);

const key = await privateKey(`${homedir()}/.cc-me.key`);
const cc = new CcMeClient({ privateKey: key });

console.log(await cc.inboxUrl());

const { requests } = await cc.claim({
  limit: 10,
  poll: true,
});

const handled = [];
for (const request of requests) {
  console.log(request.method, request.path, request.text());
  handled.push(request.id);
}

await cc.ack(handled);
